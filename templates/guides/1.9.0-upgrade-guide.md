# Couchbase Capella Provider 1.9.0: Upgrade and Information Guide

This guide covers what's new, changed, and fixed in version 1.9.0 of the Couchbase Capella Terraform Provider.

## New Features

* Manage cluster deletion protection with [`couchbase-capella_cluster_deletion_protection`](https://registry.terraform.io/providers/couchbasecloud/couchbase-capella/latest/docs/resources/cluster_deletion_protection)
* The `deletion_protection` attribute is now available on the [`couchbase-capella_cluster`](https://registry.terraform.io/providers/couchbasecloud/couchbase-capella/latest/docs/resources/cluster) resource and data source, reflecting the cluster's current deletion protection state
* Private endpoints now expose the `private_endpoint_dns` attribute on the [`couchbase-capella_private_endpoints`](https://registry.terraform.io/providers/couchbasecloud/couchbase-capella/latest/docs/resources/private_endpoints) resource

## Breaking Changes

### `tenant_id` is now required for Azure network peers

If you are managing Azure network peers using [`couchbase-capella_network_peer`](https://registry.terraform.io/providers/couchbasecloud/couchbase-capella/latest/docs/resources/network_peer), the `tenant_id` field inside the `azure_config` block is now required. Previously it was optional. Add this field to your configuration before upgrading if it is missing:

```terraform
resource "couchbase-capella_network_peer" "azure_peer" {
  organization_id = "<organization_id>"
  project_id      = "<project_id>"
  cluster_id      = "<cluster_id>"
  name            = "my-azure-peer"
  provider_type   = "azure"
  provider_config = {
    azure_config = {
      cidr            = "10.6.0.0/16"
      resource_group  = "<resource_group>"
      subscription_id = "<subscription_id>"
      tenant_id       = "<tenant_id>"   # now required
      vnet_id         = "<vnet_id>"
    }
  }
}
```

### `cors.origin` is now required when a CORS block is present on an app endpoint

When configuring CORS on [`couchbase-capella_app_endpoint`](https://registry.terraform.io/providers/couchbasecloud/couchbase-capella/latest/docs/resources/app_endpoint), the `origin` field inside the `cors` block is now required. If you have a `cors` block without an `origin`, add it before upgrading.

### `cloud_provider` is now read-only on app services

The `cloud_provider` field on [`couchbase-capella_app_service`](https://registry.terraform.io/providers/couchbasecloud/couchbase-capella/latest/docs/resources/app_service) has been changed from optional to read-only. If your configuration sets this field, remove it before upgrading:

```terraform
resource "couchbase-capella_app_service" "example" {
  organization_id = "<organization_id>"
  project_id      = "<project_id>"
  cluster_id      = "<cluster_id>"
  name            = "my-app-service"
  # cloud_provider = "aws"  # remove this line
}
```

### `couchbase-capella_app_services_cidr` data source schema has changed

The `organization_id`, `project_id`, `cluster_id`, and `app_service_id` fields on the [`couchbase-capella_app_services_cidr`](https://registry.terraform.io/providers/couchbasecloud/couchbase-capella/latest/docs/data-sources/app_services_cidr) data source are now required top-level attributes. Previously they were computed fields nested inside each list item, which meant the data source could not actually be filtered. Update your configuration to set them at the top level:

```terraform
data "couchbase-capella_app_services_cidr" "existing" {
  organization_id = "<organization_id>"
  project_id      = "<project_id>"
  cluster_id      = "<cluster_id>"
  app_service_id  = "<app_service_id>"
}
```

## Enhancements

### Stricter validation for API keys and users

The `couchbase-capella_apikey` and `couchbase-capella_user` resources now validate role values and resource IDs at plan time. Previously, invalid values were only caught by the Capella API at apply time, producing a less helpful error.

**`organization_roles`** must contain only valid values:
- `organizationMember`
- `organizationOwner`
- `projectCreator`

**`resources[*].roles`** must contain only valid values:
- `projectOwner`
- `projectManager`
- `projectViewer`
- `projectDataReaderWriter`
- `projectDataReader`

**`resources[*].id`** must be a valid UUID.

**`resources[*].type`** must be `"project"` (the only supported value).

For `couchbase-capella_apikey` specifically, two additional rules are now enforced:

- `secret` can only be set together with `rotate`. Setting `secret` alone (without `rotate`) is now rejected at plan time.
- `rotate` cannot be set when creating a new API key. It is only valid when updating an existing one.

The `sort_direction` field on the `couchbase-capella_users` data source now validates that the value is either `"asc"` or `"desc"`.

### OpenAPI-spec-driven attribute validation

Validators for many resource and data source attributes are now auto-generated directly from the Capella OpenAPI spec. This affects fields across multiple resources and catches the following categories of error at plan time rather than at apply time:

- **Numeric range**: fields with minimum or maximum values defined in the spec (e.g. node counts, memory quotas) now reject out-of-range values immediately.
- **Enum values**: fields constrained to a fixed set of strings now reject unrecognised values at plan time.
- **Composition constraints**: fields governed by `oneOf` or `manyOf` rules in the spec (for example, `provider_config` on `couchbase-capella_network_peer` requires exactly one of `aws_config`, `gcp_config`, or `azure_config`) are now enforced at plan time.

If you have any existing configurations with values that do not conform to the API spec, they will now fail `terraform plan` rather than `terraform apply`.

## Bug Fixes

* Fixed a hang where creating a [`couchbase-capella_backup`](https://registry.terraform.io/providers/couchbasecloud/couchbase-capella/latest/docs/resources/backup) resource could stall for up to 90 minutes due to a race condition in backup polling
* Fixed the `configuration_type` field on clusters being incorrectly handled as a non-pointer, which could cause unexpected plan output
* Fixed cloud snapshot backup pagination, completion detection, and cleanup after restore operations
* Resources are now removed from state when they no longer exist remotely, rather than returning an error
* Rejected private endpoints are no longer retained in state
* Fixed a crash in the `couchbase-capella_buckets` data source that produced a `Value Conversion Error` for any cluster that had buckets with `vbuckets` configured. The data source now correctly exposes `vbuckets` as a read-only attribute on each bucket
* Fixed a malformed error message in `couchbase-capella_flush` that omitted the bucket ID and misleadingly implied data had been deleted when the flush actually failed

## New Resources

* [`couchbase-capella_cluster_deletion_protection`](https://registry.terraform.io/providers/couchbasecloud/couchbase-capella/latest/docs/resources/cluster_deletion_protection) - Enable or disable accidental deletion protection on a cluster

## Cluster Deletion Protection

The new `couchbase-capella_cluster_deletion_protection` resource lets you guard a cluster against accidental deletion. When `deletion_protection` is `true`, any attempt to delete the cluster through the Capella API or console will be rejected until protection is turned off.

Destroying this Terraform resource only removes it from state; it does not disable protection on the cluster. To delete a protected cluster, first set `deletion_protection = false` and apply, then destroy.

```terraform
resource "couchbase-capella_cluster_deletion_protection" "example" {
  organization_id     = "<organization_id>"
  project_id          = "<project_id>"
  cluster_id          = "<cluster_id>"
  deletion_protection = true
}
```

For more information, see the [deletion protection examples](https://github.com/couchbasecloud/terraform-provider-couchbase-capella/tree/main/examples/deletion_protection).
